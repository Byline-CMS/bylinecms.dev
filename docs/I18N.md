# Admin Interface i18n

Companions:
- [CORE-COMPOSITION.md](./CORE-COMPOSITION.md) — the broader roadmap for how Byline composes adapter packages.
- [AUTHN-AUTHZ.md](./AUTHN-AUTHZ.md) — admin user model; carries the `preferred_locale` column described below.

> **Status: design proposal.** Nothing in this document has shipped yet. The package, the column, the server functions, and the registration API are all described in the present tense as if implemented — that's the writing-style convention this repo uses for design-first docs (see `RICHTEXT.md` for the same posture before that subsystem landed). Anywhere implementation diverges from this proposal, the doc is the thing that needs to update first.

## Overview

**Two i18n axes, one Byline, kept strictly separate.** Byline already has a **content** locale axis — documents publish in many languages, the storage layer carries per-locale field values, the read pipeline resolves a per-request locale (`apps/webapp/byline/i18n.ts` → `i18n.content`). That part is shipped and is **not what this doc covers**.

This doc covers the **interface** locale axis — what language the admin UI itself renders in. The two axes are deliberately independent: an editor in a Spanish-language admin chrome routinely edits English, French, and German content. The same `i18n` config object already has the slot for it (`i18n.interface.{ defaultLocale, locales }`), but until now nothing actually consumed it as a translation switch.

`@byline/i18n` is the package that fills that slot. It owns:

- A **translation registry** — locale × namespace → keyed messages, populated at startup from built-ins + every plugin / extension / custom field that opts in.
- An **ICU formatter** layered on `intl-messageformat` (the same library the host i18n uses).
- A **React provider + `useTranslation()` hook** for client surfaces, and a **server-side `t()` helper** for loaders / server fns.
- A **locale detection cascade** — admin user's `preferred_locale` → cookie → Accept-Language → `defaultLocale`.
- A **language-switcher server fn** that persists the chosen locale to both the admin user record and a cookie.

The package is a leaf — depends on `@byline/core` only. The actual surface area that uses it (admin shell, richtext-lexical, AI plugin, custom fields) imports the hook and registers its own translations in `apps/webapp/byline/admin.config.ts`.

## Quick reference

Each entry is the minimal shape for one task. The "Edit" line tells you which file you actually change.

### 1. Enable the bundled English admin

Default registration. Every admin shell string ships in English; no per-locale work needed yet.

**Edit:** `apps/webapp/byline/admin.config.ts`

```ts
import { defineClientConfig } from '@byline/core'
import { adminTranslations } from '@byline/i18n/admin'

defineClientConfig({
  i18n: {
    interface: { defaultLocale: 'en', locales: ['en'] },
    translations: adminTranslations({ locales: ['en'] }),
  },
  // … the rest of your client config
})
```

`adminTranslations({ locales })` reads bundled JSON files in `@byline/i18n/src/admin/` and returns the `byline-admin` namespace for each requested code. Skipping it is a hard error at startup — the validator refuses to mount the admin without at least one registered locale.

→ [Translation registration](#translation-registration)

### 2. Add a second locale

Every bundled translation lives in-package at `packages/i18n/src/admin/<code>.json`. To enable a locale, list its code in `i18n.interface.locales` and pass the same codes to `adminTranslations({ locales })`. Today's bundle ships `en` and `fr`; adding more is a one-file PR (drop a new JSON, add it to the bundle map).

**Edit:** `apps/webapp/byline/admin.config.ts`

```ts
import { adminTranslations } from '@byline/i18n/admin'

defineClientConfig({
  i18n: {
    interface: { defaultLocale: 'en', locales: ['en', 'fr'] },
    translations: adminTranslations({ locales: ['en', 'fr'] }),
  },
})
```

The `interface` block stays separate from the content-locale list — admin UI in French does not force document content into French. `adminTranslations({ locales: ['xx'] })` throws at config time when the requested code is not in `bundledLocales`.

→ [Locale configuration](#locale-configuration)

### 3. Translate a string in your own admin code

Inside any admin shell component or admin server fn output. Same hook everywhere.

**Edit:** wherever the string lives.

```tsx
import { useTranslation } from '@byline/i18n/react'

export function PublishButton() {
  const { t } = useTranslation('byline-admin')
  return <button>{t('actions.publish')}</button>
}
```

→ [The translation hook](#the-translation-hook)

### 4. Contribute translations from a plugin

The `@byline/ai` plugin ships its own namespace. Same shape as the built-in admin namespace: per-locale JSON files inside the plugin's own package, plus a factory matching `adminTranslations`'s `{ locales }` signature.

**Edit (plugin author):** the plugin's own package.

```
packages/ai/src/i18n/
├─ en.json
├─ fr.json
└─ index.ts   ← exports `aiTranslations({ locales })` factory
```

```ts
// packages/ai/src/i18n/index.ts
import type { LocaleCode, TranslationBundle } from '@byline/i18n'
import { mergeTranslations } from '@byline/i18n'

import enJson from './en.json'
import frJson from './fr.json'

const BUNDLES = { en: enJson, fr: frJson }

export function aiTranslations({ locales }: { locales: readonly LocaleCode[] }): TranslationBundle {
  return mergeTranslations(
    ...locales
      .filter((l) => l in BUNDLES)
      .map((l) => ({ [l]: { 'byline-ai': BUNDLES[l as keyof typeof BUNDLES] } }))
  )
}
```

```json
// packages/ai/src/i18n/en.json
{
  "drawer.title": "Ask Byline",
  "drawer.placeholder": "Describe what you want…",
  "errors.rate_limited": "You sent too many requests in a row."
}
```

**Edit (host author):** `apps/webapp/byline/admin.config.ts`

```ts
import { mergeTranslations } from '@byline/i18n'
import { adminTranslations } from '@byline/i18n/admin'
import { aiTranslations } from '@byline/ai/i18n'

defineClientConfig({
  i18n: {
    interface: { defaultLocale: 'en', locales: ['en', 'fr'] },
    translations: mergeTranslations(
      adminTranslations({ locales: ['en', 'fr'] }),
      aiTranslations({ locales: ['en', 'fr'] }),
    ),
  },
})
```

A plugin that doesn't have a translation for a requested locale silently filters it (the `.filter(l => l in BUNDLES)` line above). The boot validator gates against missing required namespaces at the framework level, not per-plugin. `mergeTranslations` is associative + deterministic: later sources override earlier ones at the `(locale, namespace, key)` granularity, with a dev-mode collision warning.

→ [Translation registration](#translation-registration) | [Namespacing conventions](#namespacing-conventions)

### 5. ICU formatting (plurals, dates, numbers)

ICU MessageFormat works inline — same syntax the host i18n already uses (`apps/webapp/src/i18n/translations/en.ts`).

```ts
// translations
'inbox.unread': '{count, plural, one {# unread message} other {# unread messages}}',
'doc.publishedOn': 'Published on {date, date, medium}',
```

```tsx
const { t } = useTranslation('byline-admin')
t('inbox.unread', { count: 3 })             // "3 unread messages"
t('doc.publishedOn', { date: new Date() })  // "Published on May 28, 2026"
```

→ [Message formatting](#message-formatting)

### 6. Server-side translation (loaders, server fns)

Same translation surface from a server context. Returns a pre-bound `t(key, values)` for the request's resolved locale.

**Edit:** the server fn or route loader.

```ts
import { createServerFn } from '@tanstack/react-start'
import { resolveServerTranslator } from '@byline/host-tanstack-start/i18n'

export const sendInvite = createServerFn({ method: 'POST' })
  .handler(async () => {
    const { t } = await resolveServerTranslator('byline-admin')
    return { subject: t('email.invite.subject') }
  })
```

The translator resolves the locale once per request from the same cascade the client uses (preferred_locale → cookie → Accept-Language → default).

→ [Server-side translation](#server-side-translation)

### 7. The language-switcher menu

The built-in `<LanguageMenu>` mounts in the admin chrome. It reads the registered `i18n.locales`, the current request locale, and calls the server fn that persists the new preference. No host-side wiring required.

**Default location:** admin shell top bar (`@byline/host-tanstack-start`).

If the host wants to render the same menu elsewhere — e.g. inside an Account preferences card — it can import the same component:

```tsx
import { LanguageMenu } from '@byline/i18n/react'

<LanguageMenu className="my-2" />
```

→ [The language switcher](#the-language-switcher)

### 8. Set the locale on an admin user account

The Account page exposes a "Default language" field backed by `admin_users.preferred_locale`. Toggling it both updates the user record (cross-device) and writes the cookie (immediate). When `preferred_locale` is `null`, detection falls through to cookie / Accept-Language / default — useful for users who haven't expressed a preference and want browser detection to win.

The same field is also surfaced on the admin-users list (`@byline/host-tanstack-start`) so a super-admin can pre-set a colleague's locale.

→ [Per-user locale preference](#per-user-locale-preference)

## Architecture

### The contract surface

Six things compose the present surface:

1. **`LocaleCode` / `LocaleDefinition`** — the existing types in `@byline/core` (`i18n.interface.locales`, `i18n.interface.defaultLocale`). `@byline/i18n` is the consumer that finally puts those values to work.
2. **The translation registry** — a frozen map of `{ [locale]: { [namespace]: { [key]: string } } }` produced by `defineClientConfig({ i18n: { translations } })`. Built once at startup, read-only thereafter.
3. **The `t(key, values?)` formatter** — `intl-messageformat`-backed, identical signature on client and server.
4. **The React provider + `useTranslation(namespace)` hook** — the only client-side consumer surface. Throws if mounted outside the provider.
5. **The locale resolver** — `resolveInterfaceLocale({ preferred, cookie, acceptLanguage })`. Pure function; the host calls it once per request and threads the result into the provider.
6. **The locale persistence server fn** — `setInterfaceLocaleFn({ lng })`. Updates the admin user record (if authenticated) AND the `byline_admin_lng` cookie. Cookie-only when no actor is present (e.g. login page).

```ts
// packages/core/src/@types/site-config.ts (excerpt — already shipped)
export interface BaseConfig {
  i18n: {
    interface: { defaultLocale: string; locales: string[] }
    content:   { defaultLocale: string; locales: string[] }
  }
  // …
}

// packages/core/src/@types/site-config.ts (new slot)
export interface ClientConfig extends BaseConfig {
  i18n: BaseConfig['i18n'] & {
    /**
     * Translation registry — produced by `mergeTranslations(...)` or by a
     * single `adminTranslations(...)` / plugin export. Required when any
     * Byline UI is mounted; `initBylineCore()` validates that at least one
     * locale is present and that every declared interface locale has a
     * bundle.
     */
    translations: TranslationBundle
  }
  // …
}

export type TranslationBundle = {
  readonly [locale: string]: {
    readonly [namespace: string]: { readonly [key: string]: string }
  }
}
```

`TranslationBundle` is intentionally structurally simple — it's just JSON. No functions, no React, no per-key metadata. That keeps the file format diff-friendly, importable by every translation tool that round-trips JSON, and easy for a third-party plugin to ship inside its own package. The "richer" features people sometimes want here (per-key descriptions, source-string fingerprints for translation services, plural-form metadata) belong to a future Phase 4 — they're not load-bearing for the first ship.

### The translation hook

Single contract, used everywhere:

```ts
function useTranslation<NS extends Namespace>(namespace: NS): {
  t: (key: string, values?: Record<string, unknown>) => string
  locale: string
}
```

- The hook reads the registry off context, looks up `namespace`, and returns a `t` bound to that namespace. Missing keys fall back through the chain described in [Lookup and fallback](#lookup-and-fallback).
- The returned `t` always returns a `string` — never `undefined`, never a React element. Components that need rich-text interpolation (e.g. embedded `<a>` tags inside a translated paragraph) use a separate `<Trans>` component that wraps the same formatter. Keeping the common path stringly-typed makes the 95% case easy to reason about.

The hook throws if mounted outside `<I18nProvider>`. The provider is mounted automatically by `@byline/host-tanstack-start`'s admin shell root; custom hosts mount it themselves.

### Server-side translation

`@byline/host-tanstack-start/i18n` exports `resolveServerTranslator(namespace)`. It:

1. Reads the request's resolved locale via `getAdminRequestContext()` (already exists for auth) — falls back to cookie + Accept-Language if no actor.
2. Looks up the namespace from the same registry the client uses.
3. Returns a `{ t, locale }` object identical in shape to the client hook's return.

Loaders, `createServerFn` handlers, and email templates all use the same call.

### Translation registration

Three ways to register translations, all converging on the same `TranslationBundle` shape:

**a. The built-in admin bundle.** `adminTranslations({ locales: [...] })` reads bundled JSON files from `packages/i18n/src/admin/` and returns a `byline-admin` namespace bundle for each requested code. The set of available codes is exported as `bundledLocales`; unknown codes throw at config time. Adding a new locale is a one-file PR — drop `xx.json` next to the existing files and add it to the bundle map.

**b. A plugin's exported factory.** Plugins follow the same shape — per-locale JSON files inside the plugin's own package, plus an exported factory taking `{ locales }`. The host merges via `mergeTranslations(adminFactory({...}), pluginFactory({...}))`. No side effects at module load — the host's `admin.config.ts` wires every merge explicitly.

**c. Ad-hoc inline.** A small custom field or a one-off override:

```ts
defineClientConfig({
  i18n: {
    translations: mergeTranslations(adminBundle, {
      en: { 'my-app': { 'banner.welcome': 'Welcome back' } },
      fr: { 'my-app': { 'banner.welcome': 'Bon retour' } },
    }),
  },
})
```

**Why explicit merge rather than side-effect registration.** Side-effect registration (`registerTranslations(...)` in a plugin's top-level module) creates load-order dependencies — the plugin must be imported before any UI renders, *and* its import side-effect must actually run (which dead-code-elimination can defeat in some bundlers). The explicit-merge model mirrors what `RichTextField` registration already does in this codebase: the host's `admin.config.ts` is the one file that knows about every wired-in subsystem.

### Locale configuration

Default locale + permitted set live on `i18n.interface`. One optional companion slot carries display names for the language switcher:

```ts
i18n: {
  interface: {
    defaultLocale: 'en',                   // fallback when detection produces nothing useful
    locales: ['en', 'fr'],                 // permitted set; values outside this list are rejected
    localeDefinitions: [                   // optional — display names for the switcher menu
      { code: 'en', nativeName: 'English' },
      { code: 'fr', nativeName: 'Français' },
    ],
  },
  // …
  translations: { … },                     // required when locales is non-empty
}
```

`localeDefinitions` is the host's chance to override what `Intl.DisplayNames` produces — most commonly to capitalize romance-language names (`Français` rather than CLDR's `français`). Per-code resolution in the menu is: explicit `localeDefinitions` entry → `Intl.DisplayNames(code).of(code)` → the raw code. Partial coverage is allowed: define names for the codes you care about, let the fallback handle the rest.

`initBylineCore()` validates at boot:

- Every locale in `interface.locales` has at least one namespace in `translations`. Missing → fail-fast with the missing-locale name and a pointer to `adminTranslations({ locales })`.
- `defaultLocale` is in `interface.locales`.
- Key set drift between locales surfaces as a warning (not an error) — partial translations are fine, but contributors see the gap.

### Lookup and fallback

For `t('button.publish', { count: 3 })`:

1. **Active locale** — try `bundle[activeLocale][namespace]['button.publish']`. If present, format with ICU and return.
2. **Default locale** — try `bundle[defaultLocale][namespace]['button.publish']`. If present, format and return; in dev, `console.warn` once per `(locale, namespace, key)` triple about the miss.
3. **Key fallback** — return the raw key. Loud-by-default: the user sees `button.publish` on screen, which is uglier than the English fallback but makes the gap impossible to miss during development. Hosts that prefer the cleaner UX swap this for "always fall back to default" via a build-time config flag.

### Locale detection cascade

Per request, resolved once:

1. **`admin_users.preferred_locale`** — the authenticated user's explicit choice. Wins when set.
2. **`byline_admin_lng` cookie** — set on every language switch; persists across sessions. Independent of any host-side `lng` cookie the front-end site might use (different cookie name to avoid cross-talk).
3. **`Accept-Language` header negotiation** — via `@formatjs/intl-localematcher`, matching against `i18n.interface.locales`.
4. **`i18n.interface.defaultLocale`** — last-resort fallback.

The cascade is the same on client and server — the server runs it during request resolution, the client reads it back from the provider once the page hydrates. No "locale flickering" between SSR and hydration because both paths produce the same answer.

### Per-user locale preference

The `admin_users` table gains a `preferred_locale` column:

```sql
preferred_locale text NULL  -- null = "use detection cascade"
```

Migration is additive and reversible; no backfill required. The locale resolver treats `null` as "skip this tier and continue cascading."

Surfaced in the UI in two places:

- **Account preferences** — a `Select` of `i18n.interface.locales`, plus a "Use browser default" option that sets the column back to `null`. Same component the language-switcher menu in the chrome uses, just bound to the form field instead of the server fn directly.
- **Admin users list (super-admin)** — column + edit field, so an admin can set a colleague's default before they first log in.

The server fn that updates the column also writes the cookie, so the change is visible without forcing a sign-out / sign-in cycle.

### Namespacing conventions

A namespace is a flat string. Convention:

- `byline-admin` — the built-in admin shell (chrome, dashboards, lists, forms, errors, dialogs).
- `byline-<package>` — every other Byline-shipped package (`byline-richtext-lexical`, `byline-ai`, `byline-storage-s3`, …).
- `<your-org>-<plugin>` — third-party plugins. Convention is the package name with `@` and `/` flattened; e.g. `@my-org/lexical-myfeature` → `my-org-lexical-myfeature`.

Hierarchical keys inside a namespace are dot-separated:

```
chrome.sidebar.collapse
forms.validation.required
errors.network.timeout
```

Convention only — the runtime treats keys as opaque strings.

### The language switcher

`<LanguageMenu>` (default-exported from `@byline/i18n/react`) renders a dropdown of `i18n.interface.locales`. On change:

1. Calls `setInterfaceLocaleFn({ lng })` (the server fn described above).
2. The server fn writes the cookie and — if the request carries an authenticated actor — the `admin_users.preferred_locale` column.
3. The hook refetches the provider's resolved locale and re-renders.

The menu is mounted by default in the admin chrome's top bar (between the search affordance and the user menu). Custom hosts can remove it via a `chrome.languageMenu: false` setting on `ClientConfig.admin`, or move it via the existing chrome-slot composition.

### Message formatting

`intl-messageformat` is the floor — same library the host already uses. Supports:

- Plurals: `{count, plural, one {# message} other {# messages}}`
- Selects: `{gender, select, male {…} female {…} other {…}}`
- Dates / times / numbers: `{date, date, medium}`, `{n, number, ::percent}`
- Nested formatters: `{count, plural, one {{count, number} message} other {{count, number} messages}}`

The formatter is built once per `(locale, namespace, key)` and cached for the lifetime of the registry — `intl-messageformat`'s parse step is the expensive part, and the registry is immutable, so the cache is safe.

### Bundling and code-splitting

For the first ship every locale's bundle is part of the initial admin JS payload. Two locales × ~5 kB gzipped of admin strings = ~10 kB total — well below the threshold where code-splitting earns its complexity. Because the bundle map in `@byline/i18n/admin` uses static `import enJson from './en.json'` statements, the bundler sees a fixed-size set at build time; the consumer's tree-shake can't currently drop locales the host doesn't pass to `adminTranslations({ locales })` (the lookup is dynamic).

Once a project ships more than ~5 locales the eager-bundle cost starts to matter, and a lazy-loading variant earns its complexity. See **Phase 3** below.

## Future phases

Concrete next steps, roughly ordered by likely sequence. None committed to a timeline.

### Phase 1 — Foundation (shipped)

The package, the registration API, the per-user column, the language-switcher menu, the bundled English and French admin namespaces. Additional locales arrive by adding JSON files alongside `en.json` / `fr.json` in `packages/i18n/src/admin/` — one-file PRs, no new package boundary per locale.

### Phase 2 — Growing the bundled locale set

More translations land as more contributors sign on. The flow is concrete: open `packages/i18n/src/admin/en.json` as the source, copy to `<code>.json`, translate the values, add the code to the bundle map in `index.ts`. The contract test (see Known followups: CI drift detection) catches missing keys between the new locale and the canonical English bundle. No package boundary, no separate version, no separate publish — translators iterate as fast as their PRs land.

The earlier design considered shipping each locale as its own npm package (`@byline/i18n-fr`, `@byline/i18n-de`, …). That added publishing ceremony, a per-locale build, and a workspace-package count that scales linearly with translation count, in exchange for nothing the in-monorepo file approach doesn't already deliver. Long-tail languages the core team doesn't want to maintain can still ship as third-party packages — `mergeTranslations` accepts any `TranslationBundle`, so a `@some-org/byline-i18n-fa` is a valid composition.

### Phase 3 — Lazy locale loading

Replace the eager static-import map in `@byline/i18n/admin` with a registry of async loaders. The provider awaits the active-locale loader during SSR; client-side switches suspend the shell while the new locale loads. Earns its complexity once a host ships more than ~5 locales.

### Phase 4 — Authoring-time metadata

A second registration shape that carries per-key metadata: source description, plural-form hint, deprecated-since marker, source-string fingerprint. Powers integration with external translation services (Crowdin / Lokalise / Phrase) and a future internal "missing translations" admin tile. Strictly opt-in; the simple `{ key: string }` shape stays the floor.

### Phase 5 — RTL support

Right-to-left locales (Arabic, Hebrew, Persian, Urdu, …) need more than translated strings — the admin shell's CSS needs to flip. Tailwind has a `dir-rtl:` modifier, but every absolute-positioned floating UI, every margin / padding hardcode, every icon-then-text pattern is a candidate for review. Worth scheduling as its own design pass once a community-translated RTL locale lands and surfaces the real list of breakages.

### Phase 6 — Pluralization beyond ICU

ICU plural rules cover the vast majority of languages cleanly. Some (Polish, Russian, Arabic) want all six plural categories (`zero`, `one`, `two`, `few`, `many`, `other`); the formatter supports them today. Languages with grammatical gender on adjectives that depend on the subject's gender need a `{gender, select, …}` wrapped around a `{count, plural, …}`, which works but is awkward. Revisit once a real translator hits the wall.

### Phase 7 — Inline translation editing (out of scope, here for completeness)

Some headless CMSes ship an "edit translations in the admin UI" affordance — a contributor sees `??? button.publish` on screen, clicks it, and types the translation inline. Powerful, but a significant build (writes back to bundle files, source-control round-trip, conflict handling). Not in any planned phase; tracked here so the question doesn't get re-opened ad hoc.

## Known followups

Carried into Phase 1 implementation. Each is **deferred, not speculative** — direction is decided; just not implemented yet.

### Admin-shell string extraction

Every string currently inlined in `@byline/host-tanstack-start/admin-shell/**` needs to move into the `byline-admin` namespace. A scripted pass plus manual review is the realistic shape — automated tools generate keys from English text, the manual pass collapses near-duplicates ("Cancel" vs "cancel" vs "CANCEL"). Not blocking the Phase 1 package itself; the bundle starts empty and grows as the extraction lands.

### Validation messages

Zod's built-in error messages are English-only. Two options: (a) a per-locale Zod error map registered alongside the namespace, (b) a `byline-admin.validation.<rule>` namespace the form runtime translates from before render. Option (b) keeps the dependency direction clean (the schemas don't need to import the i18n package), but loses Zod's runtime error metadata that some hosts want.

### Date / number locale separate from message locale

Today the formatter uses the active locale for both message lookup and `{date, date, medium}` formatting. Most users want both to track together. A small minority — say, a Spanish-language admin who prefers ISO-8601 dates regardless — would want them split. Not blocking; surface as a per-user setting if a request emerges.

### Custom field translations

A custom `FieldComponent` author should be able to ship translations for its own labels (help text, placeholder, validation summary). The mechanism is the same `mergeTranslations(...)` call; the host wires it during `defineClientConfig`. The only design point worth nailing is the **convention** — does the custom field's namespace match the package name, or do we let custom fields piggy-back on `byline-admin`? Recommendation: package name. Forces a small consistency tax (no opportunistic overrides of built-in keys), pays back with predictable namespace ownership.

### Server-side detection ordering when no actor is present

The login page itself has no authenticated actor. The detection cascade reduces to cookie → Accept-Language → default. Worth a deliberate decision: if a user has previously set their preferred locale and then logged out, should the cookie still win on the login screen? Recommendation: yes (the cookie was their explicit choice; sign-out doesn't revoke that). Document explicitly.

### Translation drift detection in CI

Once two or more locale bundles exist, a CI check that compares key sets between them is the standard guardrail. Pattern: enumerate every `(locale, namespace, key)` triple, ensure every namespace's key set is the same across locales. Warning, not error — partial translations are fine, but contributors should see the drift surfaced.

## Worked example — the AI plugin end-to-end

The `@byline/ai/plugins/lexical` extension already serves as the worked third-party example for the [richtext extension contract](./RICHTEXT.md#worked-example--the-ai-plugin-end-to-end). The same plugin grows an `i18n/` directory:

```
packages/ai/src/i18n/
├─ en.json
├─ fr.json
└─ index.ts   ← exports `aiTranslations({ locales })` factory
```

```json
// packages/ai/src/i18n/en.json
{
  "toolbar.button": "Ask AI",
  "drawer.title": "Ask Byline",
  "drawer.placeholder": "Describe what you want…",
  "drawer.submit": "Generate",
  "drawer.cancel": "Cancel",
  "errors.rate_limited": "You sent too many requests in a row. Wait a moment and try again.",
  "errors.context_too_long": "Your selection is too long. Trim it and try again."
}
```

`fr.json` carries the same keys, translated.

The plugin's React component:

```tsx
import { useTranslation } from '@byline/i18n/react'

function AiDrawer() {
  const { t } = useTranslation('byline-ai')
  return (
    <Drawer title={t('drawer.title')}>
      <textarea placeholder={t('drawer.placeholder')} />
      <button>{t('drawer.submit')}</button>
    </Drawer>
  )
}
```

The host wires it once:

```ts
// apps/webapp/byline/admin.config.ts
import { mergeTranslations } from '@byline/i18n'
import { adminTranslations } from '@byline/i18n/admin'
import { aiTranslations } from '@byline/ai/i18n'

i18n: {
  interface: { defaultLocale: 'en', locales: ['en', 'fr'] },
  translations: mergeTranslations(
    adminTranslations({ locales: ['en', 'fr'] }),
    aiTranslations({ locales: ['en', 'fr'] }),
  ),
},
```

No registration side-effects, no React-context registry hop, no per-plugin provider tree. The same `mergeTranslations` is what the richtext-lexical adapter, the storage providers' admin-shell badges, and any user plugin all flow through.

## Why not just adopt the host i18n outright?

The host pattern in `apps/webapp/src/i18n/` is close to what's described above — `intl-messageformat`, a namespaced bundle, a React provider, a cookie. The reason it isn't the right thing to ship as-is:

- **It targets the front-end site, not the admin.** The host's `lng` cookie carries the site-visitor's interface language. The admin needs its own cookie so editors aren't forced into the visitor's locale.
- **No per-user storage.** Admin editors expect their preference to follow them across machines; the host pattern doesn't have a place to put that.
- **No extensibility surface.** Plugins / extensions / custom fields need a registration path; the host pattern hard-codes its namespaces in one file.
- **Wrong package layer.** Putting the admin i18n in the host webapp would tie it to TanStack Start's lifecycle. The richtext / storage / admin packages all need to ship translations without depending on a specific host framework.

What the host pattern *does* establish, and what this design directly inherits: ICU-via-`intl-messageformat`, TS-module bundles (not JSON, per the Nitro asset-extension issue documented in `apps/webapp/src/i18n/translations/index.ts`), namespace-then-key structure, cookie-as-persistence-medium, `@formatjs/intl-localematcher` for Accept-Language negotiation.
