:::note[Experimental]
The CLI installs Byline into an existing TanStack Start application. It is
currently tested against up-to-date TanStack Start sites using the Nitro
(agnostic) adapter.
:::

Create a TanStack Start application with:

```sh
npx @tanstack/cli@latest create

# or

pnpm dlx @tanstack/cli@latest create
```
Select the Nitro (agnostic) adapter:
```
◆  Select deployment adapter:
│  ○ None
│  ○ Cloudflare
│  ○ Netlify
│  ● Nitro (agnostic)
│  ○ Railway
└
```
Once the TanStack Start application is ready, initialize Byline with:

```sh
npx @byline/cli@latest init

# or

pnpm dlx @byline/cli@latest init
```
Byline requires Node `>=20.9.0`. The installer supports pnpm, npm, Yarn, and Bun,
and uses the package manager declared by the workspace that owns the application.
In a monorepo, run it from the application directory: app files stay there, while
lockfiles and workspace-level settings are handled at the nearest workspace root
whose package patterns include the app. A conflicting `--pm` choice is rejected;
a noninteractive package.json-only workspace with no manager metadata must pass
`--pm`.

For pnpm, the installer creates or patches the owning `pnpm-workspace.yaml` and
merges the required build approvals rather than asking you to stop for
`pnpm approve-builds`:

```yaml
allowBuilds:
  "@google/genai": true
  esbuild: true
  protobufjs: true
  sharp: true
```

Existing YAML is preserved. A package.json-only workspace also has its existing
workspace patterns carried into the new file. Invalid YAML or a non-map
`allowBuilds` value is left for manual repair; stale
`pnpm.onlyBuiltDependencies` configuration is removed because pnpm 10+ no longer
reads it.

If there are any issues, you can follow the example application in the Byline CMS main repo under `apps/webapp`.

For AI-assisted editing, add the relevant provider API keys described in
`apps/webapp/.env.local.example`.

:::warning[Keep public and admin layouts separate]
The installer places Byline routes under the pathless `routes/_byline` layout.
Put the public application's top-level layout under a separate pathless route,
such as `routes/_frontend` or `routes/_public`. Move public styling, headers,
footers, and other layout concerns out of `__root.tsx` and into that public
layout's `route.tsx`. This prevents public application styles from affecting the
Byline dashboard.
:::

See the TanStack Router docs for [File-Based Routing](https://tanstack.com/router/latest/docs/routing/file-based-routing) and [Virtual File Routes](https://tanstack.com/router/latest/docs/routing/virtual-file-routes) for more information.

## Route prompts and generated files

`init` asks where to mount the admin UI (default `/admin`) and sign-in page
(default `/sign-in`). Admin and sign-in paths may both be nested, such as
`/internal/cms` and `/staff/login`; every segment must match
`[a-z][a-z0-9-]*`. The configured API path is resolved from
`byline/routes.ts` and may also contain multiple segments. The installer rejects
admin, API, and sign-in trees that overlap, and requires sign-in to remain
outside both configured trees. It also rejects roots that collide with locale
prefixes, `_serverFn`, `_build`, `uploads`, `static`, or `public`, plus any
segment named `index` or `route`, which is special to TanStack file routing.

The answer is applied to both `byline/routes.ts` and the physical route files and
factory IDs below `src/routes/_byline/`. For example, `/cms` produces
`src/routes/_byline/cms/...`, `/internal/cms` produces
`src/routes/_byline/internal/cms/...`, and `/staff/login` produces
`src/routes/_byline/staff/login.tsx` with route ID `/_byline/staff/login`.
Changing `byline/routes.ts` alone does not remount TanStack file routes, and
`src/routeTree.gen.ts` must be regenerated by TanStack rather than edited.

The CLI reads route and locale configuration with a restricted, non-executing
static AST evaluator. It never imports, transpiles, or runs application config.
The supported subset is literals, object and array literals, same-file top-level
`const` references and shorthand properties, static object/array spreads,
parentheses, `as const`, `satisfies`, type assertions, and the directly imported
`resolveRoutes()` helper from `@byline/core`. Only the relevant properties are
evaluated, so an unrelated dynamic property does not make an otherwise static
route or locale declaration unsafe.

The evaluator fails closed for imported values, mutable or cyclic bindings,
computed properties, property access such as `process.env`, arbitrary function
calls, methods/accessors, and mutation or effectful use of evaluated bindings.
If a relevant installed config cannot be parsed or resolved with this subset,
the routes phase plans no writes. A missing, malformed, or structurally invalid
canonical `routes.ts` template likewise blocks every route/config write rather
than generating from an uncertain source.

## Reruns and upgrades

A normal `byline init` structurally re-detects every phase, so old completion
flags cannot hide files or configuration required by a newer CLI. Complete phases
are skipped; use `--from`, `--to`, or `--only` to narrow the run, and `--force`
to rerun phases that already detect as complete. Missing generated artifacts and
recognized canonical predecessors can be upgraded, while divergent user-owned
scaffold, route, Vite, Turbo, and CI files are preserved with a manual note.

The CLI derives its dependency policy from its own release. For CLI `4.3.0`,
registry-backed `@byline/*` dependency ranges must be wholly within
`>=4.3.0 <5.0.0-0`; missing or incompatible declarations are planned at
`^4.3.0`. Unbounded ranges, ranges admitting v5, and tags such as `latest` are
not compatible. Explicit compatible `workspace:` ranges are preserved. Bare
`workspace:*`, `workspace:^`, and `workspace:~` links are never replaced: the
linked version must resolve from exactly one included workspace package manifest
and satisfy the same range, otherwise the dependency phase blocks for manual
repair.

The scaffold phase preserves existing files and defers generated route-config
migration to the routes phase. Static configuration that semantically resolves
to the prompted admin, API, and sign-in paths is considered aligned and is left
unchanged, even when it is user-authored or has a different source shape. That
semantic check does not establish generated provenance: automatic migration is
limited to an exact recognized current or previous generated `routes.ts`
predecessor.

For recognized generated installations, changing either mount atomically
creates or verifies the replacement route files, patches `byline/routes.ts`, and
removes the old generated sign-in route or admin tree as one planned change set.
Nested admin trees are generated and migrated in full. If the old tree, an
existing destination, or another coupled file contains user-owned content, the
entire coupled migration is blocked; user files are never deleted or
overwritten. An unrelated user-owned route that cannot be proven part of the
generated predecessor is preserved and reported for manual review.

Before applying a planned create, patch, or delete, the CLI checks every target
against the previewed state. If any target or migration precondition changed, no
part of that stale write set is applied. This is a stale-plan guarantee, not a
transactional filesystem rollback after an unexpected I/O failure.

NOTE: If you have manually configured Byline by copying code from the example application in the repo (byline directories, .env, start, server, __root.tsx, and vite.config.ts settings), and only want to provision the database and seed the super-admin and example docs in the new application, use `byline setup` instead of `byline init`:

```sh
npx @byline/cli@latest setup

# or

pnpm dlx @byline/cli@latest setup
```

`setup` runs preflight plus the database-provisioning and seed phases (`db` → `db-init` → `seed-admin` → `seed-docs`). It does not modify scaffold or application configuration files, but it does update `.byline-install.json` with answers and phase completion. Useful flag examples:

```sh
# Provision the DB and seed both the super-admin and example docs (default)
byline setup

# Provision the DB and seed the super-admin only
byline setup --no-seed-docs

# Provision the DB and seed example docs only
byline setup --no-seed-admin

# Provision the DB without running either seed
byline setup --no-seed-admin --no-seed-docs

# Destructive: drop and recreate the database without the reset prompt
byline setup --reset --i-mean-it

# Re-run every phase even if recorded as complete (non-destructive on its own —
# migrations re-apply as no-ops, seeds are idempotent)
byline setup --force

# Full nuke-and-pave: drop and recreate the database, then re-run every phase
byline setup --force --reset --i-mean-it
```

By default `setup` consults `.byline-install.json` and skips phases already recorded as complete. `--force` bypasses that and re-runs each phase against fresh state — useful after a manual DB reset, when you want to re-seed, or to re-verify a setup is healthy. A confirmation prompt fires before either of the `--force` flows runs (skippable with `-y`); the destructive `--reset` flow has its own confirm inside `db-init` unless `--i-mean-it` is also passed.
On an installation whose `db-init` phase is already recorded complete, combine
`--reset` with `--force`; otherwise phase detection skips the reset.

Before running any phase, `setup` performs a quick pre-flight: it bails if required
`@byline/*` packages are missing or declare an incompatible range, bails if
neither `.env` nor `.env.local` exists, and warns-and-confirms when an existing
file set is incomplete or lacks keys Byline expects (some keys may legitimately
be supplied via shell env). `setup` does not install or upgrade packages; use
`byline init` for that or for a new TanStack Start app that needs the full
scaffold.
